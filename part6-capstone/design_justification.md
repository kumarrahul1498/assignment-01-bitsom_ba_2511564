## Storage Systems

I would use a hybrid architecture because the hospital has four very different workloads. For patient history and treatment records, I would use a relational OLTP database such as PostgreSQL or MySQL. This system is best for strongly consistent patient, admission, medication, and encounter data. For monthly management reporting, I would load curated data into a cloud data warehouse where finance, bed occupancy, and department metrics can be aggregated efficiently. For the doctor-facing natural language search use case, I would add a vector database that stores embeddings of clinical notes, discharge summaries, and other text-heavy records so doctors can query history in plain English. For ICU device vitals, I would use a streaming platform plus a time-series database or scalable data lake storage, because vitals arrive continuously and must be retained in high volume. Finally, for readmission prediction, I would train the model on historical data assembled from the warehouse and selected clinical features from the OLTP and time-series layers.

## OLTP vs OLAP Boundary

The OLTP boundary ends at the systems responsible for operational care workflows: patient registration, doctor notes, prescriptions, admissions, and live ICU measurements. These systems support day-to-day transactions and must prioritize correctness, concurrency, and fast writes. The OLAP boundary begins once data is extracted, cleaned, and transformed into analytical models for reporting and machine learning. That includes curated warehouse tables for management dashboards, feature tables for readmission prediction, and document chunks plus embeddings used by the doctor query layer. In other words, the transactional system captures care events; the analytical system reorganizes those events for insight, retrieval, and prediction.

## Trade-offs

A major trade-off is architectural complexity. Using separate systems for OLTP, warehousing, streaming vitals, and vector search is more powerful, but it increases operational overhead, integration effort, and governance risk. To mitigate this, I would define a clear data contract for each source, use a centralized metadata/catalog layer, and implement automated pipelines with monitoring and lineage tracking. That keeps the architecture modular while reducing the risk of inconsistent patient data across systems.
